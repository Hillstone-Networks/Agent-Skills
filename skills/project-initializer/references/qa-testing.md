# 测试工程师规范参考

用于编写测试计划、用例、自动化脚本与测试报告时查阅；缺陷排查遵循「先根因、后修复」。

## 测试计划

| 项 | 说明 |
|----|------|
| 目标与范围 | 本阶段/版本测试目标；覆盖的功能模块、接口或端到端场景 |
| 测试类型 | 功能、接口、UI、回归、性能、安全等及占比 |
| 环境与数据 | 测试环境、账号与测试数据准备、依赖服务 |
| 进度与风险 | 与迭代/发布对齐的测试节奏；依赖、环境、人力等风险 |
| 准入/准出 | 开始测试的条件；发布准出标准（如用例通过率、P0 全过、无未关闭严重缺陷） |

## 测试用例

| 字段 | 说明 |
|------|------|
| 用例 ID | 建议格式：`{模块缩写}_{序号}`，如 `USER_001`、`API_LOGIN_02`，便于追溯与去重 |
| 标题 | 一句话说明验证点，可读且唯一 |
| 前置条件 | 环境、数据、权限等执行前需满足的条件 |
| 步骤 | 可执行的操作序列；每步对应明确动作与输入 |
| 预期结果 | 每步或整体可验证的结果；避免「正常」「正确」等模糊表述 |
| 优先级 | P0/P1/P2 或 高/中/低，与项目定义一致；P0 为核心/主流程 |
| 所属模块 | 功能或接口模块名，便于筛选与统计 |

**编写原则**：覆盖正常、边界与异常；步骤与预期一一对应；避免重复用例；新需求/变更需同步增删改用例。

## 自动化约定

- 接口测试：pytest + requests 或项目既有 API client；用例与实现分离；数据与断言清晰。
- UI 测试：Playwright/Selenium 等；页面对象或组件封装；避免裸 XPath/CSS 与重复步骤。
- 命名：测试方法 `test_{场景}_{预期}`；目录与模块/功能对应。
- 禁止：硬编码密钥/敏感信息；无断言或仅打印；依赖执行顺序的用例。

## 测试报告

执行汇总、通过/失败/阻塞统计、缺陷摘要、环境与版本、风险与建议。

## 系统性排错（根因优先）

原则：**先根因、后修复**；禁止只治标。四阶段：根因调查 → 模式对比 → 假设与验证 → 实施与验证。修复前用可复现该缺陷的用例锁定根因；修复后该用例由红变绿并跑全量防回归。过程红线：未确认根因不实施修复；多次修复失败应停止补丁式修改，视为设计/架构问题。详细流程见本技能 SKILL.md「Phase 1 问题 12」所引用的 QA 规范说明。
