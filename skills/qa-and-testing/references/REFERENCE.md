# 测试工程师规范参考

用于编写测试计划、用例、自动化脚本与测试报告时查阅。

## 测试计划

| 项 | 说明 |
|----|------|
| 目标与范围 | 本阶段/版本测试目标；覆盖的功能模块、接口或端到端场景 |
| 测试类型 | 功能、接口、UI、回归、性能、安全等及占比 |
| 环境与数据 | 测试环境、账号与测试数据准备、依赖服务 |
| 进度与风险 | 与迭代/发布对齐的测试节奏；依赖、环境、人力等风险 |
| 准入/准出 | 开始测试的条件；发布准出标准（如用例通过率、P0 全过、无未关闭严重缺陷） |

## 测试用例

| 字段 | 说明 |
|------|------|
| 用例 ID | 建议格式：`{模块缩写}_{序号}`，如 `USER_001`、`API_LOGIN_02`，便于追溯与去重 |
| 标题 | 一句话说明验证点，可读且唯一 |
| 前置条件 | 环境、数据、权限等执行前需满足的条件 |
| 步骤 | 可执行的操作序列；每步对应明确动作与输入 |
| 预期结果 | 每步或整体可验证的结果；避免「正常」「正确」等模糊表述 |
| 优先级 | P0/P1/P2 或 高/中/低，与项目定义一致；P0 为核心/主流程 |
| 所属模块 | 功能或接口模块名，便于筛选与统计 |
| 类型 | 功能/接口/UI/边界/异常等（可选） |

**编写原则**：覆盖正常、边界与异常；步骤与预期一一对应；避免重复用例；新需求/变更需同步增删改用例。

## 用例命名与组织

- **用例标题**：`[模块] 场景简述 — 预期` 或 `验证 xxx 在 yyy 条件下 zzz`。
- **目录/文件**：按模块或功能分目录；文件名与模块/接口对应，如 `user_login.md`、`order_api.xlsx`。
- **ID 分配**：同一模块内连续编号；预留扩展空间（如 10、20、30）。

## 自动化约定

| 类别 | 说明 |
|------|------|
| 接口测试 | pytest + requests 或项目既有 API client；用例与实现分离；数据与断言清晰；可配置 base_url、token |
| UI 测试 | Playwright/Selenium 等；页面对象或组件封装；避免裸 XPath/CSS 与重复步骤；等待与重试策略统一 |
| 命名 | 测试方法：`test_{场景}_{预期}`，如 `test_login_success_returns_token`、`test_list_pagination_with_invalid_page_returns_400` |
| 目录 | 与模块/功能对应，如 `tests/api/test_user.py`、`e2e/order/test_create_order.py` |
| 数据与清理 | 使用 fixture 或独立数据文件；用例间隔离；必要时清理脏数据或还原状态 |
| 禁止 | 硬编码密钥/密码/敏感信息；无断言或仅打印；依赖执行顺序的用例 |

## 测试报告

| 内容 | 说明 |
|------|------|
| 执行汇总 | 计划执行数、通过、失败、阻塞、跳过；通过率与趋势 |
| 缺陷摘要 | 按严重程度/模块统计；典型或未关闭问题简述 |
| 环境与版本 | 被测版本、环境、执行时间 |
| 风险与建议 | 未覆盖项、环境/数据限制、后续改进建议 |

## 与开发协作

- 接口测试：优先依据接口契约（OpenAPI、项目 docs）编写用例与断言；字段与错误码与后端约定一致。
- 缺陷描述：复现步骤、预期与实际、环境与数据、日志/截图；可关联用例 ID。
- 回归范围：与开发确认变更影响面；用例与代码/需求双向追溯（用例 ID、需求 ID、提交/PR）。

## 通用禁止

- 无预期结果的步骤或用例。
- 无法自动断言的主观描述（如「体验良好」）作为唯一预期。
- 在脚本或配置中硬编码生产密钥、密码、敏感数据。
- 跳过必要的环境准备、数据清理或状态还原导致用例相互污染。
